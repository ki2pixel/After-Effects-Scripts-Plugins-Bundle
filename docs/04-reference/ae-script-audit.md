# AE Script Audit Matrix — Feb 2026

**TL;DR**: Ce document est le **journal de bord** des analyses de scripts tiers. Il sert de preuve et de source brute. Une fois un pattern identifié ici, il doit être **extrait et dispatché** vers les guides techniques génériques (`coding-patterns`, `ae-internals`, `capabilities`).

## Protocole de Dispatch Documentation (New v3)

Pour éviter que ce fichier ne devienne un cimetière d'informations mortes, appliquez la règle **"Chimie vs Pharmacie"** après chaque audit :

1.  **Ici (`ae-script-audit.md`)** : C'est le **Labo**. Notez tout ce que vous trouvez (lignes de code, bizarreries). C'est la preuve brute.
2.  **`coding-patterns.md`** : C'est le **Manuel de Chimie**. Extrayez la *logique universelle* (ex: "Comment lier un calque à un Null caché") sans mentionner le script commercial spécifique dans le titre.
3.  **`ae-internals.md`** : C'est la **Pharmacie**. Stockez les *données brutes* (MatchNames, IDs, Noms de calques) dans les tableaux correspondants.
4.  **`capabilities.md`** : C'est le **Catalogue**. Listez la *capacité fonctionnelle* (ex: "Génération procédurale") permise par ces découvertes.

### Matrice de Décision

## Le problème à traiter
Vous devez enrichir la documentation interne sans analyser 500k lignes de JSXBIN à chaque audit. Aujourd'hui, aucune trace n'explique que *3D Primitives Generator* injecte `Pseudo/3DPrimitiveControlsTop-0003` dans les expressions ou qu’*Automation Toolkit* autorise l’exécution de JavaScript arbitraire sur la file de rendu. Résultat :
- ❌ Les équipes consultent `legacy-adobe/matchnames` et ne trouvent pas les suffixes propriétaires utilisés par ces scripts.
- ❌ Les correctifs doc sont ad hoc et non traçables.
- ✅ Ce rapport fournit une matrice extensible (copiez/collez une ligne par script) + recommandations ciblées pour `ae-internals.md`, `capabilities.md` et la future section « Undocumented Effects ».

## Portée de l’audit
- Couvre 90+ scripts : primitives 3D, gestion de marqueurs, sélecteurs, renaming global, info-graphics CEP, Anchor tools, AutoCircularMotion, Automation Toolkit, AW Autosaver, et toutes les références listées dans la matrice ci-dessous (copiable).
- S’appuie uniquement sur le code présent dans `/Scripts_AE/` (ligne sourcée dans les citations).
- Ne modifie pas `legacy-adobe/` directement ; on documente les écarts ici puis on pousse les ajouts dans `ae-internals.md` ou de nouveaux annexes.

## Trade-offs documentés

| Option | Avantages | Limites |
| :--- | :--- | :--- |
| **Centraliser ici (Audit Log)** | Traçabilité parfaite, citations de code, historique. | Pas lisible pour le dév qui veut juste coder. |
| **Dispatcher (Documentation v3)** | `coding-patterns` devient un vrai manuel d'architecture propre. `ae-internals` devient une base de données exhaustive. | Nécessite une discipline rigoureuse : un audit = 3 mises à jour de fichiers. |

## Golden Rule
> Documente d’abord l’écart exact observé dans un script tiers (avec citation), puis décide où il doit vivre (matchNames référentiel, pattern coding, guide CEP). Aucune observation ne doit rester seulement dans un audit ad hoc.
